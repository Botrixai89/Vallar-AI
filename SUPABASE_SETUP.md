# Supabase Auth Setup (Magic Link / Email OTP)

This app uses **Supabase Auth** for magic link sign-in. Follow these steps to finish setup.

## 1. Environment variables

Create a `.env` file in the project root (copy from `.env.example`):

```env
VITE_SUPABASE_URL=https://bqzeezvihuvietzjzlxq.supabase.co
VITE_SUPABASE_ANON_KEY=<your-anon-key>
```

- **VITE_SUPABASE_URL**: Your project URL (you already have this).
- **VITE_SUPABASE_ANON_KEY**: Get this from the Supabase Dashboard (see below).  
  **Do not** use the service role key in the frontend; the browser client must use the **anon (public) key** for security.

### Where to get the anon key

1. Open [Supabase Dashboard](https://supabase.com/dashboard) → your project.
2. Go to **Project Settings** (gear) → **API**.
3. Under **Project API keys**, copy the **anon** / **public** key (not the `service_role` key).

---

## 2. Enable Email OTP / Magic Link in Supabase

1. In the Supabase Dashboard, go to **Authentication** → **Providers**.
2. Ensure **Email** is enabled.
3. Under **Email**, enable **Confirm email** if you want new users to confirm before signing in (optional).
4. For magic link only (no password):
   - **Enable Email provider** and use the default “Magic Link” flow.  
   - No extra provider config is required; `signInWithOtp` sends the magic link.

---

## 3. Site URL and redirect URLs

1. Go to **Authentication** → **URL Configuration**.
2. Set **Site URL** to your app’s origin, e.g.:
   - Local: `http://localhost:8080`
   - Production: `https://your-domain.com`
3. Add **Redirect URLs** (one per line) so Supabase can redirect after the magic link click:
   - `http://localhost:8080/**`
   - `https://your-domain.com/**` (when you deploy)

After the user clicks the magic link in their email, Supabase redirects to your app (e.g. `/chat`). The client restores the session via the URL hash and `onAuthStateChange` fires so the app sees the user as signed in.

---

## 4. Optional: custom email templates

Under **Authentication** → **Email Templates** you can edit:

- **Magic Link** – content of the email that contains the sign-in link.

---

## 5. Flow summary

1. User enters email on landing → `signInWithOtp({ email })` is called.
2. Supabase sends the magic link to that email.
3. User clicks the link → browser opens your app (e.g. `/chat`) with tokens in the URL.
4. Supabase client reads the URL, creates a session, and `onAuthStateChange` runs.
5. App stores the session and uses `session.user.email` as `sessionId` for the VALLAR webhook.

---

## 6. Conversation history table (`vallar_conversations`)

The sidebar loads past conversations from the `vallar_conversations` table. Create it in the SQL Editor if it doesn’t exist:

```sql
create table if not exists vallar_conversations (
  id bigint generated by default as identity primary key,
  user_email text not null,
  session_date text not null,
  role text not null check (role in ('user', 'assistant')),
  content text not null,
  created_at timestamptz default now()
);

-- Optional: RLS so users only see their own conversations
alter table vallar_conversations enable row level security;

create policy "Users can read own conversations"
  on vallar_conversations for select
  using (auth.jwt() ->> 'email' = user_email);

create policy "Users can insert own conversations"
  on vallar_conversations for insert
  with check (auth.jwt() ->> 'email' = user_email);
```

- `session_date`: unique conversation id (app uses ISO timestamp, e.g. `2025-03-14T12:00:00.000Z`).
- Each row is one message; sidebar groups by `session_date` and shows the first user message as the preview.

---

## 7. Service role key (backend only)

You have `SUPABASE_SERVICE_KEY` for server-side or backend use (e.g. admin APIs, RLS bypass). **Never** put it in frontend code or in a `VITE_*` env var. The browser only uses `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`.
